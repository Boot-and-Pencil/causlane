# M07.1 — CLI explain/why

- **Stage:** S07 — Observability, explainability и DX
- **Status:** `done_or_near_done`
- **Purpose:** explain, why-blocked, why-not-parallel, graph export, replay diagnostics.

## Delivery note

M07.1 lands the CLI/DX surface for explain/why diagnostics without creating a
second scheduler or replay authority:

- `causlane explain replay --bundle <bundle.json> --trace <trace.json> [--json]`
  is a top-level alias for the existing bundle-bound `ReplayExplain` path.
- `causlane why-blocked --graph <graph.yaml|json> --op <action_id>:<op_index>
  [--json]` loads a typed graph snapshot and reports graph/frontier blockers.
- `causlane why-not-parallel --graph <graph.yaml|json> --op
  <action_id>:<op_index> --with <action_id>:<op_index> [--json]` reports
  pairwise write-scope conflicts plus each op's existing blocker reasons.

The graph snapshot is a CLI input DTO, not a persisted graph export. Full
Mermaid/DOT/JSON graph slices land in M07.2. The implementation delegates to
`ReplayTrace::verify_explain`, `GraphIndex`, `select_frontier`,
`why_not_parallel_from_index`, and `pair_conflict`; it renders their typed
answers instead of duplicating decider logic.
