## Progress (2026-06-22)

`GuardedExecutor::spend_barrier` already authorized the barrier (deny-by-default) and
derived a capability, but never re-checked that capability at the point of execution.
M06.6 adds the missing **spend-time admission** decider and enforces it at the executor
seam. Additive — `ExecutionCapability` is **not** formal-bound (`kani_target.rs` imports
`ExecutionBarrier` + `capability_binding_matches` but not `ExecutionCapability` /
`validate_for_barrier`), so the formal contour is untouched (no bundle-hash / formal-IR
regen).

- `crates/causlane-core/src/domain/capability.rs` (additive, pure):
  - `enum CapabilitySpendRefusal { NotBoundToBarrier { cause }, OpMismatch { capability_op,
    requested_op }, Expired { expires_at, now } }` and the borrow struct
    `CapabilitySpendRequest<'a> { barrier, requested_op, now }`.
  - `ExecutionCapability::spend_admits(&self, req) -> Result<(), CapabilitySpendRefusal>`
    — fail-closed, deny-wins: `validate_for_barrier` (→ `NotBoundToBarrier`) ≻ op-exactness
    (→ `OpMismatch`) ≻ `expires_at <= now` (→ `Expired`, `<=` parity with the authz gate).
    `Ok(())` means structurally bound, op-exact, and live. It reuses `validate_for_barrier`
    rather than re-deriving the structural checks.
- `crates/causlane-runtime/src/guarded_executor.rs`: `SpendError` gains
  `CapabilityRefused`; `spend_barrier` runs `spend_admits` **between** derivation and
  `executor.execute`, so the worker spends only a capability that admits this exact op
  and is live at `now`. The `Expired` check enforces the **lease-derived** expiry that
  authz never looks at (authz judges the decision's freshness, not the lease's).
- `docs/07-security-and-authz.md`: the spend-time admission seam + precedence
  (`NotBoundToBarrier` ≻ `OpMismatch` ≻ `Expired`), governed by ADR-0013.
- **Executable checks (no prose-only):** the core property test
  `spend_admits_grid_matches_oracle` exhausts the capability × barrier × requested_op ×
  temporal-boundary grid against an **independent** oracle (a single structural
  conjunction, not the early-return ladder), with non-vacuity over all four verdicts;
  plus negative controls (`spend_refuses_wrong_barrier`, `spend_refuses_forged_id`,
  `spend_refuses_op_substitution`, `spend_refuses_at_expiry_boundary` pinning the `<=`
  boundary, `spend_precedence_structural_beats_op_beats_expiry`). At the runtime seam,
  `expired_capability_is_refused_and_does_not_execute` proves the wiring: a valid Allow
  but a lease-expired capability is refused and the op never runs.

**Risk/limitation:** **additive only — the formal contour is untouched.** A freshly
derived capability is always barrier-bound and op-exact, so `NotBoundToBarrier` /
`OpMismatch` are unreachable through `spend_barrier` in-process (the only seam-reachable
refusal is `Expired`); those checks are load-bearing once a capability crosses a trust
boundary (cached/serialized/separately minted), and are fully covered at the core layer.
**Out of scope:** HMAC attestation **verification** at spend (the keyed-secret concern);
max-age policy for `None`-expiry capabilities; live lease re-verification; cross-process
capability transport/serialization; changing `ExecutorPort::execute`. No serde.
