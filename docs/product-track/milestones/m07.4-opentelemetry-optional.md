# M07.4 — OpenTelemetry optional

- **Stage:** S07 — Observability, explainability и DX
- **Status:** `done_or_near_done`
- **Purpose:** OTLP logs/traces/metrics adapter; fail-open for telemetry only.

## Delivery note

M07.4 adds an optional `causlane-runtime/otel` feature. The default runtime
build remains dependency-minimal; OpenTelemetry crates are pulled only when the
feature is enabled.

The adapter exposes `OpenTelemetryTraceSink`, a `TraceSinkPort` implementation
that consumes the M07.3 `TraceSpan` projection. It does not classify
`AuditEventKind` and does not install global OpenTelemetry providers. The sink
owns local SDK providers and can be built either from host-supplied providers or
from an OTLP/HTTP protobuf configuration using a blocking HTTP client.

For every `TraceSpan`, the sink emits:

- an OpenTelemetry span;
- an OpenTelemetry log record;
- a low-cardinality counter sample named `causlane.trace_spans`.

Causlane identifiers (`trace_id`, `span_id`, `parent_span_id`, `event_id`,
`action_id`, `plan_hash`) are preserved as attributes. The adapter does not
fabricate OpenTelemetry trace/span ids from causlane ids.

## Limitation

OpenTelemetry export remains derived telemetry. It is not replay input, not
observed truth, and not an enforcement mechanism. Exporter/backend failures are
telemetry loss only; audit append and protocol semantics stay in the existing
audit/replay/kernel paths.
