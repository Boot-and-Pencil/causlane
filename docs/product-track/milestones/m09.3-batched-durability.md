# M09.3 — Batched durability

- **Stage:** S09 — Performance, reliability и high-throughput readiness
- **Status:** `done_or_near_done`
- **Purpose:** Group commit for barriers/audit without violating write-ahead.

## Delivery note

M09.3 promotes batch audit append from adapter-only helper to the stable
`AuditLogPort` contract. The same port now exposes:

- `append` for immediate single-event writes;
- `append_batch` for all-or-nothing ordered batch writes.

In-memory, SQLite and Postgres adapters still share one batch admission path
through `AuditAppendState::prepare_batch`; no second audit API, journal, replay
schema or generated formal artifact is introduced. SQLite/Postgres batches are
transactional. `TraceProjectingAuditLog` emits derived spans only after the
authoritative append or batch append succeeds.

## Negative controls

- duplicate ids inside a batch leave adapter state unchanged;
- non-monotonic supplied indexes inside a batch leave adapter state unchanged;
- tracing emits no batch spans when audit append fails;
- SQLite batch rows persist in event order in one transaction.
