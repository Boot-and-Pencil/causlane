## Progress (2026-06-22)

The lane model is codified as a pure kernel module bound to the tier model.

- `crates/causlane-core/src/domain/lane.rs`: `Lane { lane_id, tier, capacity,
  capability, fairness_weight }` bound to exactly one `Tier`; `LaneCapacity`
  (`Unbounded` / `Bounded(n)`) with `has_room`; `accepts_capability` (exact
  match, `None` = general-purpose, no-requirement work runs anywhere).
- `lane_admits(lane, active, op_tier, op_requires) -> LaneAdmission { Admit |
  Reject(WrongTier | CapabilityMismatch | CapacityExhausted) }` runs the **tier
  authority gate first**, so a lane confers no semantic authority.
- **Executable check (no prose-only):** `lane_never_grants_cross_tier_authority`
  proves exhaustively over `Tier::ALL × Tier::ALL` (bounded over capacity / active
  / capability) that any `Admit` implies `op_tier == lane.tier` and any cross-tier
  action is rejected as `WrongTier` — a lane can never bypass tier authority.
  Non-vacuity guards that admits and cross-tier rejections both occur; plus unit
  tests (capacity bound, capability matching, each rejection cause) and a
  `LaneRejection` exhaustiveness guard.
- ADR-0005 updated with the M05.2 codification note.

**Risk/limitation:** `capability` is modelled as an exact-match class token and
`fairness_weight` as carried data — fairness *ordering* and richer capability
classes are runtime/scheduling concerns (M05.4/M05.5 + runtime), not kernel
authority. Pure, additive `causlane-core` module (builds standalone; no
reducer/verify-path change). Deeper standalone Alloy/P/Lean modeling is S10
follow-up; formal contour untouched.
