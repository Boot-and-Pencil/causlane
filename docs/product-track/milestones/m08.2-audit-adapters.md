# M08.2 — Audit adapters

- **Stage:** S08 — Runtime shell и adapters
- **Status:** `done_or_near_done`
- **Purpose:** In-memory, SQLite, Postgres append-only audit; group commit policies.

## Delivery note

M08.2 replaces the placeholder runtime audit adapter with append-only audit
adapters behind the existing `causlane_core::AuditLogPort`. No second audit API
is introduced.

The default runtime build remains dependency-minimal and includes the in-memory
adapter only. Durable adapters are feature-gated:

- `causlane-runtime/sqlite-audit` exposes `SqliteAuditLog` using `rusqlite`;
- `causlane-runtime/postgres-audit` exposes `PostgresAuditLog` using `postgres`.

All adapters use the same append admission path: event ids must be unique,
missing `event_index` values are assigned monotonically, and supplied
non-monotonic indexes fail closed. Batch append helpers prepare the whole batch
before mutating adapter state; SQLite/Postgres write the prepared batch in one
transaction.

## Authority boundary

Audit adapters persist accepted audit envelopes. They do not decide lifecycle,
authorization, witness validity, conflict safety, replay verdicts, retry policy,
or projection access. Tracing remains derived from successful audit appends and
storage failure is fail-closed for the audit append.

## Storage contract

Durable adapters store the stable audit envelope columns required for adapter
durability checks:

- `event_index`;
- `event_id`;
- `action_id`;
- `plan_hash`;
- `kind`;
- `correlation_id`;
- `causation_id`;
- `occurred_at`;
- `impact_set_hash`;
- `drain_fence_scope`.

`AuditEventKind::stable_token` is the shared kind-token mapping used by the
storage envelope. Full replay payload serialization remains a later schema
boundary; M08.2 deliberately does not introduce a second trace/event schema.

## Failure mode

Duplicate ids, non-monotonic indexes, index overflow, schema/load errors, and
insert failures return `AuditAdapterError`. If durable append fails, adapter
state is not advanced. SQLite/Postgres batch append rolls back the transaction
and in-memory batch append leaves the journal unchanged.

## Tests and controls

Runtime tests cover in-memory monotonic assignment, duplicate rejection,
non-monotonic rejection, batch all-or-nothing behavior, shared envelope
projection, SQLite schema/insert/duplicate/rollback behavior, and Postgres
DDL/insert/envelope compilation without requiring a live Postgres server.
