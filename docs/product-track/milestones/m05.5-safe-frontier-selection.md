## Progress (2026-06-22)

Safe frontier selection — the consumer of the M05.4 graph indexes — is codified.

- `crates/causlane-core/src/domain/frontier.rs`: `select_frontier(index, lanes)` takes
  the M05.4 `GraphIndex` plus a `BTreeMap<LaneId, LaneCapacity>` and returns
  `FrontierSelection { selected: BTreeSet<OpId>, rejected: Vec<FrontierRejection> }`. The
  selected set is a **conflict-free antichain within lane budget**: no two selected ops
  write the same `Scope`; every selected op is structurally ready (drawn from
  `ready_by_lane`, so all required facts already produced — the antichain / no-hard-deps
  property); per lane, already-active + selected ≤ capacity (reuses M05.2
  `LaneCapacity::has_room`); a write scope is an exclusive resource. Selection is greedy in
  `OpId` order (deterministic); each rejection carries a `FrontierBlock`
  (`LaneAtCapacity` / `WriteScopeConflict{scope, with}`) — the seed for M05.8.
- `graph_index.rs`: two additive read-only accessors — `ready_nodes()` (the candidates) and
  `active_in_lane()` (occupied capacity) — used by the selector.
- **Executable checks (no prose-only):** the load-bearing property
  `frontier_is_a_conflict_free_antichain_within_budget` exhausts a small graph/lane-budget
  space and asserts conflict-free + antichain + within-budget + valid rejection causes
  (local maximality) + determinism, with a non-vacuity guard exercising both rejection
  causes; unit tests for disjoint-scope all-select, two-pending-on-one-scope single-select,
  and a full lane rejecting its ready ops. ADR-0005 updated (Codification M05.5).

**Risk/limitation:** the conflict gate is **mutable writes** only (exact-scope match,
consistent with M05.4); read-write conflicts (a reader vs a writer) are deferred — they
need `reads` on `GraphNode`. Constraint-plane (`resolve_constraints`) freeze/token gating
is not folded into selection (needs per-op claims; applied upstream at admission, and
M05.7 wires runtime updates → frontier rebuild). Verified-merge relaxation of the
exact-scope conflict is a follow-up. Pure additive `causlane-core` (builds standalone; no
reducer/verify-path change). Deeper Alloy/Lean modeling of frontier safety is an S10
follow-up; the formal contour is untouched.
