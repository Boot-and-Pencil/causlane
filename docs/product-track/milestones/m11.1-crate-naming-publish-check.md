# M11.1 — Crate naming/publish check

- **Stage:** S11 — Public pre-alpha/bootstrap and alpha publication
- **Status:** `exists_expand`
- **Purpose:** prepare deterministic no-upload readiness and publication order;
  do not claim actual crates.io upload readiness until registry-dependent probes
  pass in the release runbook.
- **Outcome:** No-publish readiness cleared local blockers and handed off to the
  staged release evidence.

## Required outputs

- generated `docs/release/publish-readiness.md` and JSON report;
- human-maintained `PUBLISHING.md` and `docs/release/publish-all-crates-runbook.md`;
- advisory crates.io name probe command;
- explicit dependency tiers and full publish sequence;
- documented warning that names are not reserved before upload.

## Acceptance checklist

- [x] Workspace version is `0.0.1`.
- [x] Internal path dependencies carry registry-compatible versions.
- [x] Package metadata and README/license files are present.
- [x] Publication order is machine-derived.
- [x] Generated readiness remains repo-local; staged upload evidence records
  actual release execution.
- [ ] Architecture/refactor gate has no errors.
- [ ] Declared Cargo binaries all have source files.
- [ ] Staged dry-run/publish procedure has been executed for the actual release.

## Risk / limitation note

This milestone does not publish or reserve any crate name. A deterministic report
may pass while `publication_execution.status = "deferred"`. That means
repository-local readiness is clear, not that crates.io upload can be performed
without the release runbook.

Do not run `cargo publish --dry-run` for all workspace crates as one pre-upload
batch. Dependent crates need their internal registry dependencies published and
indexed first.
