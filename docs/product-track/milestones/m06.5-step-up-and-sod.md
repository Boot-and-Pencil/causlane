## Progress (2026-06-22)

A strict, fail-closed step-up / separation-of-duties / freshness layer over the M06.4
coordinate approval gate. Additive — `ApprovalRef` is non-formal-bound, so the formal
contour is untouched (no bundle-hash / formal-IR regen).

- `crates/causlane-core/src/domain/approval.rs` (shared type extensions):
  - `AssuranceLevel(u8)` — an **opaque ordered** newtype the kernel compares as
    `presented >= required` (derived `Ord`). The kernel fixes only the ordering; which
    concrete factors map to which value is host policy. No MFA-mechanism taxonomy lives
    in the kernel.
  - `ApprovalRef` gains `issued_at: Timestamp`, `expires_at: Option<Timestamp>`,
    `assurance: AssuranceLevel`; `ApprovalDenyReason` gains `SelfApproval`,
    `InsufficientAssurance`, `IssuedAfter`, `Expired`, `Stale`. The M06.4
    `classify_approval` / `approval_gate` are unchanged.
- `crates/causlane-core/src/domain/approval_stepup.rs` (new, pure):
  - `ApprovalRequirement<'a> { action, plan, impact_set, initiator, required_assurance,
    max_age, now }` — what a barrier demands of its approvals at `now`.
  - `classify_approval_stepup(...)` returns `Ok(())` **only** for an exactly-bound
    `gate.approved` that is *also* from a **distinct** actor (`actor != initiator`;
    separation-of-duties, no self-approval per `docs/07-security-and-authz.md`),
    sufficiently **assured** (`assurance >= required_assurance`; step-up), and **fresh**
    (not forward-dated / expired / stale — mirroring `classify_authz_decision`,
    ADR-0011). `approval_gate_stepup(...)` is fail-closed and accumulate-then-decide, so
    it is order-independent and sets its approve flag *only* on that fully qualified
    `Ok(())` — a lone self-approval / under-assured / stale approve can never authorize.
    Precedence: `ExplicitDeny` (deny-wins) ≻ `Approved` ≻ `SelfApproval` ≻
    `InsufficientAssurance` ≻ `IssuedAfter` ≻ `Expired` ≻ `Stale` ≻ `WrongAction` ≻
    `WrongPlan` ≻ `WrongImpactSet` ≻ `Missing`.
- **Executable checks (no prose-only):** the load-bearing property
  `approval_gate_stepup_matches_oracle_and_is_order_independent` exhausts the
  coordinate × verb × actor × assurance × temporal-boundary grid (singles + all pairs),
  asserting the gate matches an **independent** oracle (membership `.any()` queries, not
  the accumulate loop) **and** is order-independent, with non-vacuity over every
  outcome; plus negative controls pinning the safety invariant
  (`lone_self_approval_never_authorizes`, `under_assured_distinct_plus_self_is_not_approved`,
  `stepup_classifier_ok_means_fully_qualified`, `assurance_below_requirement_is_insufficient`,
  `freshness_age_boundary_is_strict`, `deny_wins_over_qualified_approve_stepup`).

**Risk/limitation:** **additive only — the formal contour is untouched.** Self-approval
is forbidden (the docs' only stated SoD requirement); **N-of-M / quorum is out of scope**
(unspecified in docs — modelling it would be guessed taxonomy). Concrete MFA mechanisms /
IdP integration are host policy (it maps factors → `AssuranceLevel`). Wiring
`ApprovalRef`'s step-up attributes through `AuditEvent` / runtime / replay is later
(additive, non-breaking). Capability spend at the barrier is M06.6; projection redaction
is M06.7. No serde.
